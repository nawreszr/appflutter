Faire évoluer le projet vers une architecture micro services complète en introduisant la

Q2 — Définir le workflow GitHub
Indication : Un workflow GitHub structuré est essentiel dans tout projet collaboratif
professionnel. L'objectif ici est de mettre en place des règles et des modèles qui
garantissent la qualité du code avant toute fusion dans les branches principales.
Configurez les éléments suivants dans votre dépôt :
Protection de branche : dans les paramètres GitHub de votre dépôt ( Settings > Branches ),
activez la protection sur main et version-3 : exigez au moins une review approuvée avant tout
merge, et interdisez les pushs directs.
Template d'issue : créez le fichier .github/ISSUE_TEMPLATE/bug_report.md (et/ou
feature_request.md ) pour standardiser le signalement de bugs et de nouvelles fonctionnalités.
Template de Pull Request : créez .github/pull_request_template.md avec des sections
prédéfinies : description du changement, lien vers le ticket Jira, checklist de review (tests
passants, documentation mise à jour, etc.).
Convention de review : rédigez dans votre README.md une section décrivant les règles de
review adoptées par l'équipe (ex. : toute PR doit être relue dans les 48h, les commentaires
bloquants doivent être résolus avant le merge, etc.).

Q3 — Ajouter le micro service grading-service
(Gestion des Notes)

Indication : Créez un nouveau projet Spring Boot indépendant nommé grading-
service . Ce micro service gère l'entité Note et communique avec le micro service

étudiant pour valider l'existence d'un étudiant avant d'enregistrer une note. Respectez la
même architecture en couches que dans la Partie 2 (controller, service, repository, entity,
dto, mapper, config).
L'entité Note comporte les champs suivants :
Champ Type Description
id Long Identifiant auto-généré
studentId Long Référence vers l'étudiant (pas de FK JPA directe)
matiere String Nom de la matière
valeur Double Valeur de la note (0–20)
Implémentez les opérations CRUD complètes pour Note , avec gestion des erreurs HTTP
standard et documentation Swagger, comme établi en Partie 2.
Q4 — Ajouter Feign, Eureka et API Gateway
Indication : Cette question introduit les trois piliers de la communication dans une
architecture Spring Cloud. Voici comment les articuler :
Eureka est le serveur de registre : chaque micro service s'y enregistre au démarrage et
l'interroge pour trouver les autres. Créez un projet Spring Boot dédié ( eureka-server )

avec @EnableEurekaServer . Chaque micro service (étudiant, notes) doit inclure spring-
cloud-starter-netflix-eureka-client et se déclarer avec un spring.application.name .

Feign est le client HTTP déclaratif : dans grading-service , créez une interface annotée
@FeignClient(name = "etudiant-service") avec une méthode pointant vers GET
/api/etudiants/{id} . Feign résoudra automatiquement l'adresse via Eureka. Utilisez ce
client dans le service pour vérifier qu'un étudiant existe avant d'enregistrer une note.
API Gateway (Spring Cloud Gateway) est le point d'entrée unique : créez un projet
api-gateway qui route /api/etudiants/** vers etudiant-service et /api/notes/**
vers grading-service . Les clients externes (mobile, frontend) n'appellent plus que l'API
Gateway.

- id: grading-service

- Path=/api/notes/**

Q5 — Mettre à jour l'application mobile pour
afficher les étudiants par département
Indication : L'application mobile (Flutter ou React Native) développée en Partie 1
affichait une liste plate d'étudiants. Mettez-la à jour pour d'abord récupérer la liste des
départements via GET /api/departements , puis permettre à l'utilisateur de sélectionner
un département et d'afficher uniquement les étudiants de ce département. Toutes les
requêtes doivent désormais passer par l'API Gateway (plus directement par le micro
service).
En Flutter, utilisez un DropdownButton pour la sélection du département et un
FutureBuilder pour charger les étudiants de manière asynchrone. En React Native,
utilisez un Picker (ou FlatList de boutons) et useEffect pour déclencher le fetch à
chaque changement de département sélectionné.
Q6 — Créer un frontend Next.js pour la gestion des
étudiants et des départements
Indication : Créez un projet Next.js dans un dossier frontend/ à la racine du dépôt. Ce
frontend doit permettre de consulter, créer, modifier et supprimer des étudiants et des
départements via l'API Gateway. Organisez le projet en pages distinctes : /etudiants
pour la liste et le formulaire d'ajout/édition, /departements pour la gestion des
départements.
Utilisez fetch ou axios avec les Server Components de Next.js (App Router) pour les
lectures, et des Client Components ( "use client" ) pour les formulaires interactifs.
Stylez l'interface avec Tailwind CSS pour un rendu propre et responsive. Toutes les
requêtes pointent vers l'API Gateway.

Q7 — Mettre à jour le docker-compose.yml pour

Indication : Le docker-compose.yml à la racine du projet doit désormais orchestrer

déclarer ses dépendances avec depends_on . Veillez à ce que eureka-server démarre en
premier, puis les micro services métier, et enfin l'API Gateway et le frontend.
